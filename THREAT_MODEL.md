# Threat model

## Scope and security objective

This model covers the static public site, browser-local synthetic workflow, checked-in generated evidence, developer CLIs, optional model/sandbox adapters, repository dependencies, and CI release controls. The objective is to keep external effects subordinate to reviewed policy, confine the one real file-modification slice to repository-owned fixtures, prevent sensitive material from entering public artifacts, and make evidence tampering or control failure visible.

It does not claim that browser storage resists the visitor, that Docker defeats a hostile host/kernel, that optional cloud providers are trusted by default, or that point-in-time scans prove future safety.

## Assets and trust boundaries

Protected assets include source and fixture integrity, developer credentials, capability/policy/context definitions, approval decisions, the disposable workspace boundary, validated command results, trace/evidence bindings, the lockfile and SBOM, and the truthfulness of public claims.

Trust transitions occur between:

- static content and visitor-controlled browser state;
- synthetic issue/context text and policy/tool evaluation;
- the controller and the local Docker daemon or optional E2B service;
- the local gateway administrator credential and a scoped per-run credential;
- repository source, package registries/scanner databases, and CI;
- generated evidence and the public viewer.

## Threats, controls, and residual risk

| Threat                                       | Current repository control                                                                                                                                                                                                                                       | Residual or production requirement                                                                                                                    |
| -------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Prompt injection**                         | Context is treated as untrusted data; records have source, sensitivity, stage/persona/agent scope, state, TTL, and explicit include/exclude reasons. Tool authorization is independent of model/context content.                                                 | Add ingestion sanitization, trust scoring, model-specific defenses, monitoring, and incident response before using untrusted external sources.        |
| **Malicious issue text**                     | Browser fixtures and CLI issue input are checked-in synthetic data. The sandbox accepts no visitor issue, prompt, repository, patch, or command input.                                                                                                           | A production intake service needs content-size limits, encoding normalization, DLP, tenant authorization, and safe rendering/logging.                 |
| **Path traversal**                           | Normalize and resolve paths, reject absolute/parent traversal and `.git`, compare against exact approved targets, and test traversal variants.                                                                                                                   | Dedicated worker isolation and defense against platform-specific path/TOCTOU behavior remain necessary.                                               |
| **Unexpected file writes**                   | Snapshot before/after trees, exact allow-list enforcement, changed-file classification, and fail on any unexpected path.                                                                                                                                         | Use immutable base images, read-only source mounts plus a narrow writable layer, and server-side policy in production.                                |
| **Symlink escape**                           | Reject symlinks in controlled paths and verify real paths remain inside the disposable workspace.                                                                                                                                                                | Stronger OS primitives and race-resistant open/write APIs are required against a malicious concurrent process.                                        |
| **Secret exfiltration**                      | Public fixtures contain no credentials; browser has no credential input or live adapter; Docker validation uses no network and receives no secrets; trace/evidence fields are allow-listed and redaction-tested.                                                 | Managed short-lived secrets, workload identity, enforced egress, DLP, rotation, and provider retention review are required.                           |
| **Untrusted dependencies**                   | Exact npm lockfile, clean install, dependency review, high/critical audit gate, license inventory, CodeQL configuration, and generated SBOMs.                                                                                                                    | Registry/maintainer compromise and advisories published after the scan remain possible; enable branch protections and signed provenance verification. |
| **Sandbox escape**                           | Fixed commands, disposable workspace, no Docker socket/home/project-root mount, network disabled, non-root user, read-only root/mounts, dropped capabilities, no-new-privileges, and CPU/memory/PID/time/output limits.                                          | The Docker daemon, host kernel, image supply chain, and local account remain trusted. Use hardened dedicated workers for adversarial multi-tenancy.   |
| **Stale state**                              | Upstream redo marks dependent stages stale; context/policy/record hash or freshness changes invalidate packs; resume revalidates all bindings.                                                                                                                   | Durable transactional invalidation, authoritative clocks, and source-change events are required across distributed workers.                           |
| **Unauthorized approvals**                   | Fail-closed scope intersection, deny precedence, requester binding, separation of duties, expiry, canonical argument/target/context hashes, and single-use resume.                                                                                               | Authenticate humans/workloads, authorize server-side, and store decisions transactionally in append-only/WORM storage.                                |
| **Evidence tampering**                       | Versioned schemas, canonical SHA-256 digests, file/output/tree hashes, context and trace bindings, index/Markdown validation, and build-time evidence checks.                                                                                                    | Hashes are not signatures. Add signing, trusted timestamps, immutable retention, independent verification, and key governance.                        |
| **Supply-chain compromise**                  | Redacted Gitleaks worktree/history scans, ESLint/repository policy, npm audit, digest-pinned tools/bases, exact three-image Trivy gates and SBOMs, LiteLLM Cosign/package-floor verification, license policy, suppression expiry/use, and SHA-pinned CI actions. | Scanner/runner/database compromise and blind spots remain; the local derivative and reports are not signed attestations and no SLSA level is claimed. |
| **Poisoned or over-broad context**           | No autonomous learning; deterministic retrieval; revoked/stale/unauthorized/over-budget records are visible exclusions; episodic records require explicit policy.                                                                                                | Shared stores need moderated ingestion, signed provenance, tenant isolation, quarantine, rollback, and deletion/retention controls.                   |
| **Confused deputy or scope escalation**      | Effective authority is the intersection of persona, delegation, stage, agent, tool, resource, and path constraints; discovery grants no authority.                                                                                                               | Use audience/resource-bound tokens, workload attestation, signed delegation, and revocation.                                                          |
| **Policy downgrade or registry tampering**   | JSON Schema plus content-hash validation, lifecycle approval, exact version/hash binding, and stale-manifest rejection.                                                                                                                                          | Signed bundles, trusted distribution, monotonic version policy, and rollback protection are required.                                                 |
| **Trace leakage**                            | Safe attribute registry, bounded values, credential/multiline redaction, category-only commands/errors, and no prompts/source/output/arguments.                                                                                                                  | Central telemetry needs access control, encryption, retention, regional policy, and continuous DLP.                                                   |
| **Budget or repair-loop bypass**             | Versioned content-hashed budgets; exact pre-operation counts; measured duration checks; warn/stop events; failure evidence; non-zero stop exit.                                                                                                                  | Distributed execution needs durable reservations, atomic counters, queue cancellation, billing reconciliation, and organization kill switches.        |
| **Gateway master-key or scoped-key leakage** | Master credential exists only in local environment; live adapter is loopback-only; raw scoped key stays in gitignored recovery state; aliases contain no personal data; finally revokes; output leakage tests scan logs/evidence/build.                          | Use OS/managed secret stores, workload identity, rotation, memory hardening, short-lived token exchange, and alerts.                                  |
| **Fallback drift and provider retention**    | Allowed aliases and fallback order are registry-hashed; catalog is preflighted; each actual attempt is traced; no prompt/response is evidence; live profile is explicit.                                                                                         | Central routing change approval, provider contracts, zero-retention options, residency/DLP review, and deletion procedures are required.              |
| **Browser-state misrepresentation**          | Demo mode and synthetic labels are persistent; reset is deterministic; only harmless preferences persist; local approval state is disclosed as local and resettable.                                                                                             | Browser state is not shared truth. Production decisions need authenticated server-side state and audit ownership.                                     |

## Abuse cases intentionally removed by architecture

The public site exposes no anonymous runner, arbitrary prompt, arbitrary Git URL, patch upload, raw command, gateway endpoint, provider key form, or Docker/localhost connection. Therefore an ordinary visitor cannot use the site to execute code or spend provider funds. Adding any such endpoint would cross the current trust boundary and require a new threat model and explicit authorization.

## Verification and ownership

Tests exercise traversal and symlink rejection, allow-list enforcement, unexpected files, timeouts, cleanup, failed/successful evidence, hash tampering, context TTL/revocation/scope/budgets, unauthorized/self approval, replay and expiry, trace redaction, budget stops, model fallback/key revocation, reset, and stale-state guards. `npm run security:supply-chain` adds point-in-time repository and image evidence.

Security-sensitive changes require review of this document, [SECURITY.md](SECURITY.md), [docs/sandbox-security-model.md](docs/sandbox-security-model.md), [docs/trace-data-handling.md](docs/trace-data-handling.md), and [docs/release-evidence.md](docs/release-evidence.md). Release exceptions must use the dated, expiring suppression contract; prose alone cannot turn a blocked scanner into a pass.

## Explicit non-claims

- No production-readiness, penetration-test, formal-verification, SLSA, or zero-vulnerability claim is made.
- E2B and live LiteLLM/provider paths were not validated without credentials.
- CodeQL is configured but is not claimed as executed until hosted CI evidence exists.
- The checked-in release summary is sanitized recorded evidence, not a signed attestation.
