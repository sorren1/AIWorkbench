## Changes from v1.0.8

### Browser stability

- Firefox runs with one worker across supported environments to bound aggregate browser pressure without retries.
- Responsive coverage keeps every public route and width while reporting each route/viewport pair independently.

### Interruption-safe local tooling

- A shared lifecycle coordinator propagates cancellation, runs cleanup once in reverse registration order, bounds cleanup operations, and preserves conventional signal exit codes.
- Local sandbox runs persist exact ownership records before provider acquisition and expose guarded recovery for Docker and E2B resources.
- Model-gateway leases register revocation before vending work, serialize concurrent cleanup, and retain failed cleanup state for explicit retry.

### Commit-bound deployment evidence

- A manually dispatched GitHub workflow validates the live generated binding and canonical identity before running the existing Production browser suite.
- Desktop and mobile Lighthouse checks cover the portfolio, Workbench case study, and Workbench demo.
- A strict sanitized summary is retained for 90 days; detailed browser and Lighthouse artifacts remain restricted to 14 days.

### Supply-chain policy

- Suppression entries require exact package/image selectors, unique finding selectors, and review dates that do not follow expiry dates.
- The scanner preflight requires an available Linux Docker engine before any release evidence can pass.
- Narrow npm overrides move `@hono/node-server` and `fast-uri` to patched versions after new registry advisories appeared.
- A read-only hosted workflow can run the full scanner record gate and retain the generated summary when the release operator's machine lacks Linux Docker.

### Documentation

- Public release documentation now distinguishes immutable v1.0.8 history from reusable pre-release procedure and generated current-state evidence.
- Version labels are aligned on 1.0.9 across the package, lockfile, manifest, case-study footer, and demo fixture.

## Evidence model

The exact audited source commit must pass local checks to the extent supported by the release machine and the complete hosted Quality, dependency-review, and CodeQL gates. The hosted scanner workflow must then rerun the full supply-chain record gate against that same source and bind the exact successful zero-finding CodeQL run. Its downloaded summary becomes the only changed path in one direct evidence child. Annotated tag `v1.0.9` points to that child.

A deployment is accepted only when its build-generated binding proves the release tag, audited parent, evidence/deployed child, hosted CodeQL source and result, and verified relation. The post-deployment workflow must independently observe the same tuple at the canonical origin and pass browser, accessibility, network, header, cache, and desktop/mobile Lighthouse controls.

## Functional, simulated, and measured boundaries

- The public case study, deterministic browser demo, local workflow state machine, bounded toy-repository MCP fixture, Docker local sandbox, and loopback model-gateway tooling remain functional within their documented environments.
- Jira, GitHub, AI, Oracle, deployment, review, and enterprise MCP-style actions shown in the browser remain simulated.
- E2B and live LiteLLM/provider paths remain credential-gated and are not promoted to validated capabilities by this release.
- The static deployment does not establish Production reliability, an SLA, authenticated or shared identity, durable multi-user state, real external integrations, or safe multi-tenant execution.
- No professional adoption, revenue, performance, or Production capability claim is added.
