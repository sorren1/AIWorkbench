## Changes from v1.0.7

### Stable deployment metadata

- `/.well-known/security.txt` is now generated with the same validated `SITE_CANONICAL_URL` used by HTML metadata, `robots.txt`, and the sitemap.
- Preview builds omit `Canonical` from `security.txt`; they cannot retain or invent a Preview identity.
- A configured Production build derives `Canonical` as `https://tylerwilhite.dev/.well-known/security.txt` and rejects malformed, credential-bearing, non-HTTPS, path-bearing, query/fragment, port-bearing, and `vercel.app` origins.
- Contact, policy, language, and expiry remain stable tracked disclosure fields.

### Nested 404 behavior

- The authored 404 document uses a root base plus root-relative stylesheet, logo, case-study, and demo URLs.
- Deployment coverage visits nested missing paths with and without trailing slash/query state, requires the real 404 status and authored content, verifies the logo/style rendering, and follows both recovery links.

### Coherent public release identity

- `package.json`, `package-lock.json`, `public/site.webmanifest`, the static case-study footer, and the demo footer now expose `1.0.8` consistently.
- A deterministic test rejects drift among those version surfaces.
- README, changelog, architecture, deployment, evidence, planning, baseline, presenter, dependency-review, and hosting records now distinguish immutable v1.0.7 evidence from v1.0.8 release-specific evidence and the intended Production origin.
- Dependabot version proposals are restored with bounded queues after the public-history publication audit: five npm pull requests and two GitHub Actions pull requests. Proposals remain review inputs and do not bypass the exact-history, dependency-review, CodeQL, or complete quality gates.
- Public-history validation distinguishes those provider-created review refs from release lineage. The exact Dependabot/GitHub identity pair and discarded owner/GitHub squash form are recognized only outside `HEAD` and annotated tags; entering release lineage still requires the account-specific owner identity.

### Personal-domain route boundary

- A static portfolio index occupies the intended Production origin without presenting the Workbench as the personal home page.
- The case study, technical article, interactive prototype, assets, and recorded evidence remain one multi-page static application under `/workbench/`.
- Exact Vercel page-to-file and resource rewrites map that namespace to the existing static output. This avoids re-routing a wildcard capture through the legacy public paths, preserves direct refreshes plus page-relative assets/capability cards/evidence, and leaves unknown Workbench paths as real 404 responses.
- The build emits the portfolio home as the physical root document and copies the case study, demo, and article into physical `/workbench/` page paths. This keeps provider filesystem precedence from replacing the intended personal home with the Workbench.
- Permanent redirects move the former hosted `/demo/`, article, and `/index.html` page URLs into the namespace without adding an SPA fallback or changing the simple local-development routes. Vercel's named wildcard syntax applies immutable caching only to Vite-hashed assets.
- Metadata and the sitemap distinguish the root portfolio URL from Workbench URLs. `SITE_CANONICAL_URL` accepts the HTTPS origin only; page metadata owns the `/workbench/` paths.

The audited v1.0.8 source commit contains no `public/security/release-summary.json`. After the exact source passed local and hosted gates, the release process created one direct child whose only changed path is that generated summary. The annotated `v1.0.8` tag points to the evidence child. The deployed binding proves all of the following:

- release tag `v1.0.8`;
- audited source commit equals the evidence commit's direct parent;
- deployed commit equals the tagged evidence commit;
- hosted CodeQL analyzed the audited source and recorded zero release-blocking findings; and
- the tag/source/evidence/deployment relationship is verified.

The v1.0.8 [generated release summary](https://tylerwilhite.dev/security/release-summary.json) and [generated deployment binding](https://tylerwilhite.dev/security/deployment-binding.json) are the current authoritative records. ## Functional, simulated, and measured boundaries

Functional repository behavior remains the static case study/article, strict-TypeScript React demo, deterministic workflow state and guards, browser-local synthetic approval/review state, checked-in evidence validation, developer-invoked fixed Docker sandbox, and deterministic offline model-gateway adapter.

Every browser-visible Jira, GitHub, LLM/provider, database, enterprise MCP, deployment, hosted validation, production identity, shared approval, provider billing, remote review, notification, user, issue, repository, branch, pull request, check, log, duration, and external response remains synthetic or recorded-only as labeled. The public site exposes no execution endpoint, credential, arbitrary repository, arbitrary patch, or arbitrary command input.

Measured values are limited to named repository checks and the checked-in local Docker recording. Synthetic fixture values and policy ceilings are not measurements. Evidence hashes detect changes; they are not signatures, attestations, trusted timestamps, or proof of Production operation.

## Credential-gated limitations

- E2B is implemented behind an explicit provider selection, but no `E2B_API_KEY` was available for a live provider run. Fake contract coverage is not cloud-run evidence.
- The loopback LiteLLM/model-provider path is implemented, but no live master key/provider credential was available. The public label remains `gateway implemented; live provider path not validated`.
- Docker evidence remains local, fixture-only, and dependent on the host kernel and daemon. It is not proof of safe multi-tenant untrusted-code execution.
- Browser-local persona and approval state is inspectable and resettable, not authenticated, shared, transactional, append-only infrastructure.
- The first-party source remains without a granted reuse license; third-party components retain their own terms.

## Verified release result and limits

For the exact v1.0.8 lineage, hosted Quality and CodeQL completed for the audited source and the evidence child. The generated summary was added in that one-file child, the annotated tag points to the child, and the same commit is deployed at the Production origin. The hosted deployment binding records the exact relation and the Workbench is published at `/workbench/`. Any Vercel Preview used by the release procedure remains Preview and is not the Production evidence cited here.

The completed static deployment does not establish an availability or reliability level, authenticated production identity, shared or durable multi-user state, real Jira/GitHub/model/database/MCP operations, safe multi-tenant execution, live E2B validation, or live LiteLLM/provider validation. Those capabilities remain conditional or out of scope exactly as described above.
