# AI Delivery Workbench 1.0.0 release notes

Status: **independently audited local release; not deployed or pushed in this phase**.

## Release thesis

Version 1.0.0 presents AI Delivery Workbench as an independent public case study of the controls around coding agents: versioned capabilities, deterministic context selection, scoped tools, human approvals, runtime/model policy, budgets, traceability, and validation evidence.

The release contains two public surfaces: a static case study/article and a separately loaded interactive React workbench. One developer-only CLI demonstrates a fixed real file modification and validation path against repository-owned synthetic code. The website displays the resulting checked-in evidence; it never executes code for visitors.

## Verifiable capabilities

- deterministic browser workflow, scenarios, reset, stale propagation, deep links, exports, and accessible guided tour;
- strict-TypeScript domain guards for verification, review, validation, release readiness, authorization, and registry lifecycle;
- inspectable agent/tool capability cards and context-pack include/exclude/freshness/digest records;
- human-approval request binding, separation of duties, expiry, invalidation, and single-use local resume;
- controlled Docker patch/build/test slice with path, symlink, write-set, network, resource, timeout, and cleanup controls;
- trace/evidence/source/tested-tree/context/policy binding and explicit budget/accounting labels;
- cross-browser, axe, coverage, build/link, CSP, bundle, Lighthouse, credential, supply-chain, SBOM, and license gates.

## Evidence basis

- **Measured:** current source/build/test outputs, recorded local Docker command durations and results, changed-file set, hashes, diff, trace, budget counters, exact zero model usage/cost, bundle sizes, and Lighthouse audits.
- **Estimated:** context token estimates and any optional accounting record explicitly carrying an estimate basis.
- **Synthetic:** every browser persona, issue, repository, branch, pull request, check, log, duration, metric, and simulated provider/external-system response.
- **Professional context:** only the separately disclosed approximately 50-story statement; it is not attributed to this prototype.

## Simulated integrations

Jira, GitHub, LLM/provider generation, database access, external enterprise MCP, deployment, hosted validation, production authentication/authorization, shared approvals, remote review, and notifications are simulated in the public browser. Angular, .NET, Oracle, Jira, and GitHub are one illustrative adapter stack rather than a required environment.

## Known limitations

- No public backend or live execution endpoint exists.
- Browser-local state and approvals are not shared, authenticated, or tamper-resistant.
- Docker isolation trusts the local daemon, kernel, base image, and developer account.
- Evidence hashes are integrity checks, not signatures, attestations, or trusted timestamps.
- E2B and live LiteLLM/provider paths are implemented but were not live-validated because credentials were unavailable.
- CodeQL is configured but remains unvalidated until a hosted workflow succeeds.
- No production reliability, adoption, agent-quality, cost-performance, or security-certification claim is made.
- Canonical deployment, contact, résumé, and analytics links remain absent until real public configuration is supplied.
- At audit time, the remote production branch still contained the preserved original prototype; publishing the audited commit and tag is a separate, explicitly excluded action.
- Vite preview serves its application fallback for an unknown path. The checked-in `404.html` must be verified with the selected static host's real not-found routing before publication.
- GitHub-hosted CodeQL, dependency review, and the quality workflow have no candidate run because this release was not pushed.
- `npm ci` reports deprecation notices from transitive development-tool dependencies even though the locked vulnerability audit reports zero vulnerabilities.

## Release audit and publication boundary

The local source, build, test, browser, evidence, performance, and security results are recorded in [the independent 1.0.0 release audit](../release-audit-1.0.0.md). Before publication, the owner must still observe hosted CI, verify the selected static host applies the emitted headers and not-found routing, configure and inspect canonical metadata, and complete the ignored private publication review. This phase does not deploy or push the project.
